# OKR ダッシュボード SaaS 外販戦略 (v2)

最終更新: 2026-05-14

このドキュメントは、社内専用ツールとして運用してきた OKR ダッシュボードを SaaS 製品として外販するための戦略・モジュール構造・料金プラン・実装ロードマップを集約したものです。LP / 営業資料 / 開発判断のすべての参照点とします。

---

## 1. 製品の本質

これは「タスク管理ツール」ではなく、**「タスク中心の業務運用 OS」** である。

キャッチコピー: **「社員が今日何をすれば良いか迷わない」**

Notion や Asana のような点のツール、Microsoft 365 や Google Workspace のような点の集合体ではなく、**1 日の業務フロー全体を設計した線のツール**として位置づける。

### 思想 (絶対にブラさないこと)

1. 仕事の全てはタスク管理に集約される。タスクを正確に作れて実行できる人が優秀。
2. 差がつくのは **「タスクを作る精度」**。実行はメンタルなので技術ではない。作る精度は訓練可能なスキル。
3. タスクの発生源は 3 つ: **会議 / 思いつき / 目標 (OKR) の逆算**。発生源は分かれていても、実行者には 1 つのビューに集約されるべき。
4. 経営目線のツール (財務・OKR) をいくら磨いても、社員が自走しないと数字は動かない。現場の 1 日の積み重ねが結果を生む。

### 現実への対応 (重要)

理想は「タスクは OKR 起点で考えられる」ことだが、運営段階の実態として:
- 一般社員はほぼ OKR 起点で考えられない。思いつきをメモのようにタスクとして書く
- 管理職は OKR 起点で考え、KA を誰にアサインするかという視点を持つ

→ 製品は **両方の入口** を持つ。タスクは独立して作れるが、後付けで KA に紐付けることも、管理職が KA からアサインすることもできる。これは現状コードで既に実現されている (`ka_tasks` の `ka_key` / `report_id` が nullable)。

## 2. ポジショニング

| 呼び方 | カテゴリ | 価格相場 | 競合 |
|---|---|---|---|
| 「タスク管理ツール」 | コモディティ | 〜1,500 円/人 | Notion / Asana / Todoist |
| **「業務運用 OS」「自走する組織を作るツール」** | 空白地帯 | 数千円/人 | (比較対象なし) |

- 「タスク管理ツール」として売らない。機能比較表で埋もれる。
- 比較対象がいない領域で戦う。価格は社員一人当たり月額数千円を正当化できる設計に。安売りしない。
- デモは機能羅列ではなく、**ある企業の 1 日を物語として見せる** (朝の始業から夕方の終業まで)。
- 「ツール + 朝会運用テンプレート」のパッケージで提供。**運用ごと売る**。

## 3. ターゲット

- **規模**: 30〜200 人の成長企業
- **モデル**: B2B 営業ドリブン (初期導入費を取る前提)
- **意思決定者**: 経営者 (社員の自走力を上げたい / 朝会文化を作りたい・維持したい / タスク管理が属人化している、を感じている層)
- **最小起動単位**: 1 人法人 OK (個人版という別プランは作らず、システム上は 1 ライセンス目から法人プランで起動できる)

### 外す
- 大企業 (入れ替えコスト高)
- 零細 (仕組み化ニーズ薄)
- 個人課金市場 (Todoist / TickTick とは戦わない)

### 主戦場の規模感
20 人以上で実質 ARPU が ¥2,000〜3,000 / 人月 のレンジに乗る。10 人以下は初期導入費が割高に見えるので、当面は 20 人以上を商談優先層に置く。

## 4. モジュール構造

### 4.1 コア (Free 1 ヶ月 / 全プランで有効)

「これが無いと製品が成立しない」分割不可の機能群。1 人から成立し、2 人目以降でキラー機能が自動有効化される。

| 機能 | 内容 | 実装状況 |
|---|---|---|
| タスク管理 | KA 紐付け任意のタスク作成・編集・状態管理 | ✅ 実装済 |
| 始業強制 | 平日始業時に「今日やること」を最低 1 件登録しないと開始できない | ✅ 実装済 (`MyPageShell.jsx`) |
| 振り返り強制 | 昨日終業時の KPT 未入力なら今日始業できない | ✅ 実装済 |
| 朝会 (2 人目から有効) | ファシリテーター進行 + ステップフロー (個別報告 → 確認事項 → ネクストアクション) | ✅ 実装済 |
| 週次 MTG (2 人目から有効) | KR / KA / タスク / 確認事項を段階的に進行 | ✅ 実装済 |
| 共有・確認 (2 人目から有効) | 朝会・週次 MTG から発行、属人化解消装置 | ✅ 実装済 (`member_confirmations`) |
| OKR ライト | 個人レベルの目標・KR 設定。階層・全社サマリーなし | ✅ 実装済 (現状コードを feature flag で絞る) |

### 4.2 Standard モジュール (有料プラン)

| モジュール | 内容 | 実装状況 |
|---|---|---|
| Google 連携 | Gmail / Calendar / Drive の per-user OAuth 連携 | ✅ 実装済 |
| 会議連携 | Notion 議事録インポート / Slack 通知 / Calendar イベント連動 | ✅ 実装済 |
| AI 相談 | Claude API を用いた壁打ち / 提案 / 要約 | ✅ 実装済 (`AIPanel.jsx`) |

### 4.3 Add-on モジュール (個別月額 + 導入サポート必須)

| モジュール | 内容 | 実装状況 |
|---|---|---|
| OKR フル | 組織階層 (経営 / 事業部 / チーム) / 親子 OKR / 全社サマリー / 四半期アーカイブ / AI フィードバック / チームサマリー | ✅ 実装済 |
| プロジェクト管理 | マイルストーン (`milestones`) | ✅ 実装済 |
| CFO ナレッジベース | 経営層向け AI チャット (将来 Add-on 化候補。当面は neo-fukuoka 専用) | ✅ 実装済 |

### 4.4 neo-fukuoka 専用 (feature flag で隠す / 他テナントには見えない)

| 機能 | 理由 |
|---|---|
| PortalPage | 外部リンク集 (営業 DB / コミュニティ等) が完全に neo-fukuoka 固有 |
| WorkforceTab | 6 ロール固定 (営業 / 運営 / CS / 企画 / 総務 / PR) が業種特化 |
| CFO ナレッジベース | 将来 Add-on 化したら昇格、それまでは neo-fukuoka 専用 |
| NEO 福岡固有テキスト | AI prompt 内の自社紹介文 / Drive デフォルトフォルダ名 等 |

### 4.5 削除対象 (Phase B で消す)

- CSV インポート (`/csv`, `CsvPage.jsx`) — デモ専用
- bulk 登録 (`/bulk`, `BulkRegisterPage.jsx`) — β段階では UI で十分
- `/api/fix-orphan-okr` — 一回限りのデータ修復
- `/api/restore-ka-carryover` — 一回限りのデータ修復
- `supabase_demo_humano_*.sql` — デモ生成用

## 5. 料金プラン

| プラン | システム上の価格 | 営業側で扱うもの |
|---|---|---|
| **Free (1 ヶ月トライアル)** | ¥0 / 1 ヶ月 | クレカ登録不要、商談前提 |
| **Standard** | **¥1,980/人月** | 初期導入費 (個別見積、規模・カスタマイズに応じ 15〜80 万円幅) |
| **+ OKR フル アドオン** | 別途月額 (仮 ¥980/人月) | 導入サポート費 (個別見積) |
| **+ プロジェクト管理 アドオン** | 別途月額 (仮 ¥980/人月) | 導入サポート費 (個別見積) |

### 価格設計の根拠

- 月額 ¥1,980 は Notion ビジネス (¥2,250) より明確に下、Slack ビジネス (¥850) より上 のレンジ
- 初期導入費を載せると **30 人組織で実質 ¥2,810/人月、100 人で ¥2,230/人月** → 「業務運用 OS」として妥当
- 規模が大きいほど割安に見える構造 = 100 人超企業に売りやすい

### トライアル運用

- クレカ登録なしで開始
- トライアル中は **Standard 全機能を開放** (Free だけ見せると課金動機が弱まる)
- AI 相談はコスト抑制のため利用回数制限あり
- 1 ヶ月以内に商談 → 契約書ベースで初期費 + 月額

### β顧客の特例

- 初期費は減額 or 無料
- 月額は 50% OFF 程度
- 引き換えに **導入事例公開 / インタビュー協力** を契約条件

## 6. 差別化の核

競合 (Notion / Asana / Monday) は「タスクのリストを見せる」ところで止まっている。本製品は:

1. **「今日何をやるかを決める」ところまで肩代わりする** = 始業時の AI 提案 5 件 + 強制入力 + 振り返り
2. **「タスクを作る精度」を装置で強制する** = 粒度警告 / 曖昧動詞検出 (Phase 1 で新規実装予定)
3. **朝会機能** = 組織文化として朝会を持つ日本企業に直接刺さる
4. **共有・確認機能** = 属人化解消、ブロッカー可視化

---

## 7. やってはいけないこと

1. 社内向けと外販向けで **コードベースを分岐させる** (両方死ぬ)
2. 外販を意識した機能追加で **思想をブラす** (顧客の要望に流されない)
3. 「タスク管理ツール」カテゴリの価格に引きずられる (Notion と比較されない価格帯を維持)
4. **機能数で売ろうとする** (思想と運用で売る)

## 9. β投入までのロードマップ

戦略で打ち出した機能の **95% は既に実装済み**。残作業は主にモジュール化と差別化機能の追加。

### Phase B: 自社固有前提の汎用化 / 削除 (1〜2 週)
- 層ラベル (経営 / 事業部 / チーム) を `organizations.level_labels` で設定駆動化
- 工数 6 ロール固定の削除 (WorkforceTab を flag 隠蔽)
- NEO 福岡固有テキスト (`app/api/ai/route.js` の自社紹介文 等) を grandfathered で条件分岐
- CSV / bulk / fix-orphan / restore-ka / Humano デモ系を削除
- Notion DB ID マッピングの per-org 化を完成 (一部実装済み)

### Phase C: モジュール化 (2〜3 週)
- `organizations.enabled_modules` JSONB 追加 + マイグレーション
- `organizations.plan` カラム追加 (`free_trial` / `standard` / `standard_plus`)
- `lib/featureFlags.js` 新設 (`useFeatureFlag(key)` / `isModuleEnabled(orgId, key)`)
- 各 UI 箇所への feature flag 組み込み (下記 4.2-4.4 のモジュールキーで制御)
- neo-fukuoka はデフォルトで全モジュール ON (既存運用に影響なし)
- 「全モジュール OFF」テナントで動作検証

### Phase 1: 差別化機能の新規実装 (1〜2 週)
- 粒度警告: タスク作成時に「3 時間以上は警告」「30 分未満は分割推奨」
- 曖昧動詞検出: 「検討する / 確認する / 進める / 対応する / やる」等を弾く or 書き直しを促す
- 「今日やる」AI 提案 5 件 + 理由併記 (既に AI Panel 基盤あり、提案ロジックを追加)

### Phase D: β顧客投入 (2〜4 週)
- β顧客 1〜3 社 (うち最低 1 社はコールド経由)
- 計測指標: 粒度違反率 / 曖昧動詞使用率 / OKR 紐付け率 / 朝会開催率 / 始業強制達成率
- フィードバックループ → 製品改善

### 合計
**4〜6 週間で β投入可能** な状態。本格ローンチ (Phase 4: LP / 価格表 / 導入事例) は β 終了後 (3 ヶ月目以降)。

## 10. 計測すべき指標 (Phase 1 と並行)

### プロセス指標 (= ツールが直接動かしている挙動 = ROI の説明材料)
- 粒度違反率 (3 時間以上タスクの割合)
- 曖昧動詞使用率 (タスクタイトルに含まれる割合)
- OKR 紐付け率 (KA 紐付けタスクの割合)
- 朝会開催率 (週あたり開催日数)
- 始業強制達成率 (平日のうち始業した日の割合)
- 共有・確認発行件数 / 未解決件数

### 業績指標 (= 補助証拠、因果は弱いので主張材料には使わない)
- タスク完了数
- OKR 達成率
- 朝会所要時間
